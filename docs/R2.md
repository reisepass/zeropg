# zeropg on Cloudflare R2 (Track B)

Bringing zeropg up on **R2 + a scale-to-zero Cloudflare compute**, mirroring
what we proved on GCS + Cloud Run. R2 is the cost-optimal home: **$0 internet
egress**, cheap storage ($0.015/GB-mo), and a generous always-free tier (10 GB +
1M Class A writes + 10M Class B reads / month — see [COST-MODEL.md](../COST-MODEL.md)).

> **Status, honestly.** This VM has no Cloudflare/R2/AWS credentials and no
> wrangler, so nothing here has executed against real R2. The transport is built
> creds-optional and the **CAS conformance suite is the part validated now** —
> against GCS, which passes all probes. Everything R2-live is gated on creds and
> clearly marked *pending real R2* below.

## 1. `R2BlobStore` — the transport (TODO B1)

`packages/blobstore/src/r2.ts`, a sibling to `gcs.ts`, implements the full
`BlobStore` interface against **R2's S3-compatible API with AWS SigV4 signing**
(plain `fetch` + `node:crypto`, no aws-sdk). R2 also exposes Workers bindings,
but an S3-API transport is what runs from Node and from our experiment harness;
a binding-backed variant (`env.BUCKET`) is a later drop-in.

| method | S3 mechanism | notes |
|---|---|---|
| `get` | `GET`, optional `Range` | returns `null` on 404; etag = S3 `ETag` (quotes stripped) |
| `put` | `PUT` | conditional via `If-None-Match: *` / `If-Match: <etag>` |
| `head` | `HEAD` | etag + content-length, or `null` |
| `delete` | `DELETE` | idempotent (S3 returns 204 even when absent) |
| `list` | `GET ?list-type=2` | ListObjectsV2, paginated by `continuation-token`, XML parsed |
| `getStream` | parallel ranged `GET` | **pinned to a version via `If-Match`** (see below) |
| `putStream` | single PUT *or* multipart | small streams collapse to one PUT; large use S3 multipart |
| `copy` | `PUT` + `x-amz-copy-source` | server-side, no bytes through the client |

**Instantiation is env-var driven and creds-optional.**
`R2BlobStore.fromEnv(prefix?)` / `r2OptionsFromEnv()` read `R2_ACCOUNT_ID`,
`R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET` (or standard `AWS_*`
plus `R2_ENDPOINT`/`AWS_ENDPOINT_URL_S3`), returning `null` when creds are
absent so callers skip cleanly. The endpoint defaults to
`https://<accountId>.r2.cloudflarestorage.com`, path-style.

### Conditional-PUT semantics + retry discipline

The conditional PUT is the **only strong primitive the whole design needs**.
R2/S3 map our two preconditions directly:

- `ifNoneMatch: true` → `If-None-Match: *` (create-if-absent).
- `ifMatch: <etag>` → `If-Match: <etag>` (compare-and-swap).

On precondition failure R2 answers `412 Precondition Failed` (we also treat
`409` as a precondition failure), which the transport raises as
`PreconditionFailedError` — the single signal the lease and commit protocols
are built on.

Retries copy `gcs.ts`'s discipline **exactly**: clean `429`/`5xx` rejections
retry with jittered backoff; a thrown `fetch` (ambiguous network failure) on a
conditional PUT is **never** retried, because a retried-but-already-landed CAS
would read back as a spurious 412 → a false `FencedError` upstream. This is the
discipline R2's own conditional-write bugs (workers-sdk #6411, workerd #2572)
make non-optional.

### Version pinning on `getStream` (no generation query param)

GCS pins each ranged GET to an object *generation* via a query param, so a
concurrent overwrite can never interleave two versions' bytes mid-restore. R2/S3
has no such param; the transport pins instead with **`If-Match: <etag>`** on
every range — if the object changed underneath the restore, the ranged GET
returns `412` and we fail loudly rather than splice a torn snapshot. (In zeropg
this never fires in practice: snapshot objects are immutable and named per
generation; pinning is defence in depth.)

### `putStream`: single-PUT vs multipart

S3 PUT needs a `Content-Length`, so unbounded streaming uses **multipart
upload**: the stream is buffered into `PART_BYTES` (8 MB) parts so peak memory
stays O(part size) — the same "never buffer the whole archive" property the GCS
chunked upload has. Streams that end inside the first part collapse to a single
conditional `PutObject` (one request). **The conditional precondition is applied
to the final commit step in both cases** (the single PUT, or
`CompleteMultipartUpload`), so the commit stays atomic. *Pending real R2:* that
R2 honours `If-None-Match`/`If-Match` on `CompleteMultipartUpload` specifically
is asserted from the S3 contract and must be confirmed by the conformance suite
once creds exist (the suite already exercises conditional `putStream`-shaped
writes through `put`; a multipart-specific probe is the one gap).

### Cost model

`R2_COST` pins the table with a review date and sets `internetEgressGbUsd: 0`,
`freeTier: { 10 GB, 1M writes, 10M reads }`, and **`casStrength: 'etag'`** (next
section). It omits `maxWritesPerObjectPerSec`: R2 publishes no hard per-object
write-rate cap (Workers-side limits dominate), so — unlike GCS — there is **no
forced group-commit pacing** on R2. The manifest CAS still serialises the single
writer, which is the real rate limit anyway.

## 2. CAS strength tier: `generation` vs `etag` (TODO B3)

A new field on `CostModel`, `casStrength: 'generation' | 'etag'`, records how
strong each backend's conditional-write primitive is. **GCS = `'generation'`,
R2/S3 = `'etag'`.**

- **`'generation'` (GCS `ifGenerationMatch`)** compares a server-assigned,
  strictly monotonic *generation number*. A generation is **never reused**, so
  there is no ABA window: a token you hold is either the current version or it
  is in the past — never "the past, reincarnated".
- **`'etag'` (S3/R2 `If-Match`/`If-None-Match`)** compares an *ETag* — a content
  hash (or opaque per-version id). Unique per write in practice, but **not a
  monotonic counter**, so there is a *theoretical* **ABA window**: write A
  (etag `X`) → overwrite to B → overwrite back to a byte-identical A (etag `X`
  again). A stale `If-Match: X` that slept through B could then succeed against
  what it believes is the original A.

### Does the ABA window matter for zeropg? — Largely no, and the numbered-manifest plan closes it entirely.

The ABA hazard only bites a CAS site that **reuses an object name with content
that can revert to an earlier, identical value**. Neither zeropg CAS site does:

1. **The lease.** `lease.json` is create-if-absent, then CAS'd to a strictly
   **incrementing fencing token**. The body never reverts to an earlier value —
   the token only goes up — so even on an etag store the lease body that a stale
   writer holds can never reappear. The conformance suite's
   **fencing-monotonicity** probe asserts exactly this (token advances by 1,
   never reused) and is the per-backend gate for it.

2. **The manifest.** Today's single CAS-swapped `manifest.json` is the one place
   an etag *could* in principle recur (a commit that produced byte-identical
   manifest content). In practice the manifest embeds the fencing token, the LSN,
   and a timestamp, so identical content is already vanishingly unlikely. But the
   clean fix is already on the roadmap: **numbered immutable manifests**
   (ROADMAP v2 #1 / Track A #2) — `manifest/00000000000042.json`, highest id
   wins, every commit a **create-if-absent on a fresh, never-reused name**.
   *Create-if-absent on a name that is never reused has no ABA window on any
   etag store* — there is no prior version to revert to. So the numbered-manifest
   plan makes ABA moot on R2 **by construction**, and the only reason `'etag'`
   is tracked at all is to (a) gate the conformance suite and (b) keep the
   single-`manifest.json` mode honest until the numbered scheme lands.

**Verdict:** R2's etag CAS is sound for zeropg given the incrementing-lease
invariant and the numbered-manifest commit shape. The `casStrength` field exists
so policy can *know* the difference (e.g. prefer numbered manifests on `'etag'`
stores) rather than assume generation-grade semantics everywhere.

## 3. CAS conformance suite (TODO B2) — the per-backend kill criterion

`experiments/cas-conformance.ts` generalises E0's GCS probes into a
**transport-agnostic** suite: it takes any `BlobStore` and runs five probes that
are each a kill criterion — fail one and the lease/manifest design does not hold
on that backend. R2 has shipped genuine conditional-write bugs, so this tests
**the primitive, not the docs**.

| probe | asserts |
|---|---|
| P1 create-if-absent exclusivity | second create on a live key fails; first writer's value preserved |
| P2 compare-and-swap | CAS on current advances the version; CAS on a **stale** version fails |
| P3 20-way concurrent race | exactly one winner per round — for **both** create-if-absent and CAS, ×50 rounds |
| P4 fencing-token monotonicity | N contenders race a CAS counter; token advances by exactly 1 each round, **never reused** (the ABA guard) |
| P5 read-after-write consistency | a winning write is immediately visible |

Run: `npx tsx experiments/cas-conformance.ts`
- **GCS runs always** (creds present here), under a throwaway timestamped prefix
  `cas-conformance/<ts>/`, cleaned up after.
- **R2 runs iff creds are in the env**; otherwise it is skipped with a clear
  message — a one-liner away from being live. Results append to
  `results/cas-conformance.jsonl` with the backend name and `casStrength`.

### Verified now — GCS (`results/cas-conformance.jsonl`)

```
backend=gcs  casStrength=generation
  ✅ createIfAbsent        (0 failures)
  ✅ compareAndSwap        (0 failures)
  ✅ race20way             (0 failures)   20 writers × 50 rounds, create + CAS
  ✅ fencingMonotonicity   (0 failures)   10 contenders × 12 rounds, token 0→12
  ✅ readAfterWrite        (0 failures)
  ✅ gcs PASSED — conditional writes are a correct CAS.
```

This proves the **suite itself is correct** and that GCS passes. It is also the
regression gate for any future change to either transport.

### Pending real R2

`backend=r2` logs `verdict=SKIPPED, reason=no creds` here. The moment
`R2_ACCOUNT_ID`/`R2_ACCESS_KEY_ID`/`R2_SECRET_ACCESS_KEY`/`R2_BUCKET` are set, the

## 4. Compute target: Durable Object tier recommended (TODO B4/B5)

**Recommendation: ship on the Durable Object tier first; keep the Generic
Worker tier as the portable fallback.**

### Why the DO tier is soonest *and* safest

- A Durable Object id maps to **one globally-unique, single-threaded instance**.
  That is "exactly one writer" handed to us by the platform — so the zeropg
  lease becomes **belt-and-suspenders** rather than load-bearing. We still keep
  it (it fences a stale instance across a migration/restart window, and it is the
  *same* proven code path), but the platform already gives the property Cloud Run
  only *approximates* with `max-instances=1`.
- **R2 is the generation store underneath; the DO is the single-writer compute.**
  The DO holds the PGlite instance and the `ObjectStoreFS`, persisting to R2 —
  the layering DESIGN.md 4.7 calls for.
- **Lifecycle hooks fit zeropg's durability modes directly:** DO **alarms** are
  the relaxed/`sleep`-mode flush timer (replacing Cloud Run's idle-flush
  backstop), and **`waitUntil`** flushes on eviction. `blockConcurrencyWhile`
  serialises the cold-open so concurrent requests to a freshly-woken DO all wait
  on one restore.

Skeleton: `examples/cloudflare-do/worker.ts` (typechecks; **not deployed** — no
creds). It wires `R2BlobStore` into a `ZeroPGDurableObject`, with the alarm and
`waitUntil` flush points stubbed where `ZeroPG.open`/`flush`/`close` slot in.

The **Generic Worker tier** (plain Worker → R2, lease doing the real work) is
still wanted for the portable S3/GCS-from-Workers story, but it is *more* exposed
to the lifecycle hazards E4 probed on Cloud Run (it has no single-instance
guarantee), so it is the fallback, not the lead.

### How the experiment harness ports (E2c / E3 / E4)

Reuse the same JSONL result format; the probes are transport-agnostic where they
can be (the conformance suite already is).

- **E2c (incremental round-trip + crash matrix).** The commit path is identical
  — WAL segment PUT + manifest CAS — so E2c ports by swapping the store for
  `R2BlobStore`. *Difference vs GCS:* no `maxWritesPerObjectPerSec`, so the
  group-commit pacing that GCS forces is unnecessary on R2 (the manifest CAS
  still serialises the writer). The crash-matrix (SIGKILL at each fault point →
  reopen byte-identical) needs a Workers analogue of process-kill: force DO
  eviction / throw mid-commit and re-fetch.
- **E3 (cold-start distribution).** Workers/DO cold start differs from Cloud
  Run's ~2 s container floor — DO wake is typically much faster, so the restore
  pipeline (parallel ranged GET → gunzip → untar) should dominate end-to-end at
  every size, not the platform floor. **Seed fresh databases from the prebuilt
  empty-datadir snapshot** (as on GCS) so `initdb` never runs on the request
  path — important because Workers enforce **CPU-time limits** that `initdb` (and
  a large `restore`) can brush against. Measure the wake → restore → PGlite-open
  split exactly as E3 does on Cloud Run.
- **E4 (lifecycle).** DO **eviction** replaces Cloud Run revision-switch/SIGTERM;
  the flush-on-eviction must ride `waitUntil`. **Workers CPU-time limits** are the
  new hazard class (vs Cloud Run's ~10 s SIGTERM grace): a flush that exceeds the
  CPU budget must be avoided by keeping the first-write WAL re-baseline async (the
  README's "off the request path" property carries over directly). Zombie fencing
  is *belt-and-suspenders* here because the DO is single-instance, but the test
  still runs the lease to prove the same fencing invariant.

## 5. Free egress unlocks the read-replica / CDN-seeded client story (TODO B6)

R2's **$0 internet egress** is the structural reason to prototype the
client-hydration story on R2 first. On GCS/S3, a browser or edge replica fetching
snapshot + WAL bytes straight from the bucket pays per-GB egress (or needs a CDN
in front); on R2 it is **free**. That makes two things free that are expensive
elsewhere:

1. **`ZeroPGReplica` hydration from the bucket** — leaseless followers that poll
   the manifest and pull segments converge at zero egress cost, so you can run
   many of them.
2. **Browser-side PGlite booting from a CDN-cached snapshot** — serve the
   immutable snapshot object through Cloudflare's CDN (cache-friendly: snapshots
   are immutable and named per generation) and a browser boots a read-only
   Postgres from it for free. The manifest poll is a handful of Class B ops.

This is the read-mostly, idle-most-of-the-day target use case the README
describes, with the egress line — the one cost that scales with *readers* —
zeroed out. **Prototype here first** once the writer works live on R2.
